01

Project Overview

ScanMitra is a real-time slot booking and queue management platform for diagnostic imaging centers (ultrasound, X-ray, MRI). Patients browse nearby centers, view reviews and live queue availability, book a slot, and receive smart notifications about their queue position and any delays. Doctors and center staff use a dedicated dashboard to manage the queue, mark scans as done, and update delays in real time.

There is no payment step. After selecting a slot the patient sees a simple Booking Done Successfully confirmation screen.

Core problem solved: Patients at diagnostic centers currently wait hours without any visibility into the queue. ScanMitra gives them live queue position, ETA, and proactive delay alerts — removing the anxiety and the wasted waiting time at the center.

Recommended Tech Stack

Patient Frontend

Progressive Web App using Next.js 14 (App Router). Installable on mobile without a separate React Native app — avoids the app store overhead for a first version. Tailwind CSS + shadcn/ui for consistent, rapid UI.

Next.js 14 Tailwind CSS shadcn/ui PWA (next-pwa) React Query
Doctor Dashboard

Next.js web app running on the same codebase under a separate route group (/doctor). Protected by role-based middleware. Uses the same design system as the patient side.

Next.js 14 Role-based auth Recharts Socket.IO client
Backend API

Node.js with Express (or Fastify for better performance). Handles all REST endpoints. Runs as a separate service from the Next.js frontend so it can maintain persistent WebSocket connections — Vercel serverless functions do not support long-lived sockets.

Node.js Express / Fastify Socket.IO server JWT auth Zod validation
Queue Engine

Postgres-based job queue — a lightweight job_queue table replaces Redis/BullMQ entirely. A Node.js cron worker polls it every 30 seconds for due jobs (no-show timers, reminders, ETA recalculations). No extra infrastructure needed — Supabase Postgres is already there.

Postgres job table node-cron Socket.IO broadcast
Database

PostgreSQL via Supabase — free tier includes 500MB database, built-in auth, and file storage all in one place. Prisma ORM for type-safe queries and migrations. Supabase also replaces the need for a separate Redis instance for live queue state.

Supabase Postgres Prisma ORM
Notifications

Web Push API (native browser standard, completely free) for push notifications — no Firebase needed. Nodemailer with Gmail SMTP for transactional email (booking confirmation, report ready). No paid services at all.

Web Push API web-push (npm) Nodemailer + Gmail
File Storage

Supabase Storage (free tier: 1GB) for scan reports (PDF), profile images, and center photos. Already included in the Supabase free plan — no separate service needed. Files are served via signed URLs with expiry for security.

Supabase Storage Signed URLs 1GB free
Deployment

Vercel for the Next.js frontend. Render free tier for the Node.js backend (supports persistent WebSocket connections, 750 free hours/month). GitHub Actions for CI/CD. Everything runs on free tiers.

Vercel (frontend) Render (backend) GitHub Actions

Queue Engine Logic

Step-by-step flow

1. Booking created — Status set to PENDING. Confirmation SMS/push sent to patient. Token number assigned sequentially per center per day.

2. Day-of confirmation — 1 hour before slot, an automated reminder is sent. Status moves to CONFIRMED.

3. Slot time reached — BullMQ job fires. Status set to IN_QUEUE. Patient's queue screen goes live via WebSocket. Doctor dashboard shows the patient as next-up.

4. Patient arrives & is called — Doctor clicks "Call next." Status moves to IN_PROGRESS. A scan duration timer starts internally to detect overruns.

5. Scan done — Doctor clicks "Mark Done." Status moves to DONE. Next patient in queue is automatically promoted. A "Leave Review" prompt is sent to the completed patient.

No-show (30-minute buffer)

Buffer logic: When a patient reaches IN_QUEUE status but has not been marked as arrived within 30 minutes, a BullMQ job fires. The system checks the next patient in line and sends them a WebSocket + push notification: "The patient ahead is running late — do you want to come in earlier?" If they accept, the queue cascade-advances and all downstream ETAs are recalculated. If they decline or don't respond within 5 minutes, the queue order remains unchanged. The late patient is marked NO_SHOW only if the doctor explicitly confirms it.

Delay broadcast

Delay flow: The doctor clicks "Mark Delay" and enters the delay in minutes. The system adds this delay to the base ETA of every upcoming booking for that day and fires a batch of push notifications: "Your appointment at City Scan Center is now estimated at 3:45 PM (30-min delay)." The Queue Status screen updates live via WebSocket for all connected patients.

Postgres job queue (replaces Redis/BullMQ)

A job_queue table in Supabase Postgres stores all scheduled jobs. A Node.js cron (node-cron) runs every 30 seconds on the backend, picks up due jobs, executes them, and marks them done. This is 100% free and requires no extra infrastructure.

# job_queue table columns
id UUID PK
type String # NO_SHOW_CHECK | REMINDER | ETA_BROADCAST
payload JSON # { bookingId, centerId, ... }
runAt DateTime # when to execute
status String # PENDING | DONE | FAILED
createdAt DateTime

Live queue state

Instead of Redis, current queue state (current token, delay minutes) is stored directly in a queue_state table in Postgres, updated on every doctor action. The Queue Status screen reads this on connect, then receives all further updates via WebSocket — so database reads are minimal.

Out of Scope (V1)

The following are deliberately excluded from this version as per project requirements:

No online payment integration. After slot selection, patients see a simple "Booking Done Successfully" screen. Payment is handled in person at the center — it is outside ScanMitra's scope entirely.

Also excluded from V1: center analytics dashboard, multi-doctor per center scheduling, insurance integration, DICOM viewer for scans, and a native iOS/Android app (PWA covers mobile for V1).
